# Supabase 擴展決策與時程規劃

## 目前方案：Nano（免費）

| 資源 | 限制 | 目前用量 | 預估上限 |
|------|------|---------|---------|
| 資料庫 | 500 MB | < 1 MB | 即使 3 年資料也不會超過 10 MB |
| 月活躍用戶 | 50,000 MAU | < 100 | 遠低於限制 |
| API 請求 | 無硬性限制 | ~100/天 | 巡檢每 15 分鐘 + 少量前端讀取 |
| Realtime 連線 | 500 同時 | 0（尚未啟用） | 即使啟用也不會超過 10 |
| Edge Functions | 2M 呼叫/月 | 0（未使用） | — |

## 何時需要升級

### 不需要升級的情況（維持 Nano）

- Repo 數量從 15 增加到 100 以下
- 每天巡檢 96 次（每 15 分鐘）
- 每天數十人瀏覽首頁
- 累積數千筆 issue 紀錄

### 考慮升級到 Pro（$25/月）的時機

| 觸發條件 | 說明 |
|---------|------|
| **Nano 暫停機制** | Nano 方案閒置 7 天會暫停專案，需手動恢復。若巡檢因此中斷，考慮升級 |
| **需要自動備份** | Nano 無每日自動備份。若資料重要性提高（如對外展示、正式營運），升級以獲得備份保障 |
| **需要自訂域名 SSL** | 若前端需要用自訂域名存取 Supabase API |
| **需要更多 Realtime** | Pro 提供更多同時 Realtime 連線 |

### 不太可能需要的升級

- **Team（$599/月）**：多人協作管理 Supabase，本專案不需要
- **Enterprise**：合規、SLA、專屬支援，完全不適用

## 擴展策略

### 短期（現在 ~ 3 個月）

- 維持 Nano 免費方案
- 監控 Supabase Dashboard 的用量統計
- 若遇到暫停問題，在巡檢腳本加入健康檢查（先 ping Supabase，若無回應則提示）

### 中期（3 ~ 12 個月）

- 若會員數量顯著增加（> 50 人），評估是否升級
- 若需要 Realtime 功能（第四階段），確認 Nano 的 Realtime 限制是否足夠
- 考慮啟用 Supabase Edge Functions 取代部分巡檢邏輯

### 長期（12 個月以上）

- 若 AIPMTW 發展為正式營運平台，升級到 Pro 確保穩定性和備份
- 評估是否需要將更多功能（如申請/刪除空間）也遷移到 Supabase

## 成本效益分析

| 方案 | 月費 | 年費 | 適用時機 |
|------|------|------|---------|
| Nano | $0 | $0 | 現在，且可能持續很長時間 |
| Pro | $25 | $300 | 正式營運、需要備份、或暫停問題影響巡檢 |

**結論**：以目前的規模和使用模式，Nano 方案綽綽有餘。暫無升級需求，但應注意閒置暫停機制。
