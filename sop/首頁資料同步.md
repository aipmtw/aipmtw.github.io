# SOP：首頁資料同步

## 目的

確保 aipm.com.tw 首頁顯示的資訊與各 repo 實際狀態一致，包括卡片資料、巡檢結果、需求紀錄等。

## 現況架構

目前所有資料硬編碼在 `index.html` 中：

| 資料區塊 | 說明 | 更新時機 |
|---------|------|---------|
| 卡片 HTML | 15 張成員卡片（名稱、commit、時間） | 每次巡檢同步 |
| `issueData` | 各 repo 最近 3 筆已完成 issue | 每次巡檢更新 |
| `repoNames` | repo ID → 顯示名稱對照 | 新增/刪除 repo 時 |
| `checkResult` | 巡檢結果摘要 | 每次巡檢更新 |

## 同步項目

### 1. 卡片資訊

每張卡片包含：
- **最新 commit hash**：`gh api repos/aipmtw/<repo>/commits?per_page=1`
- **commit 時間**：UTC 轉台北時間（+8）
- **有效提交數**：Collaborator 提交的 issues 總數

### 2. 新增 Repo 偵測

- 比對 public repo 清單與現有卡片
- 排除系統 repo（如 space-template）
- 自動新增卡片 HTML + issueData + repoNames + checkResult.details

### 3. 刪除 Repo 清理

- 用 `gh api repos/aipmtw/{name}` 檢查每張卡片的 repo 是否存在
- 404 → 自動移除卡片和相關 JS 資料

### 4. 需求紀錄（issueData）

- 查詢每個 repo 最近 3 筆已關閉 issues
- 記錄 author、completed（台北時間 ISO 格式）、summary（繁體中文）

### 5. 時間戳記

| 欄位 | 說明 | 更新規則 |
|------|------|---------|
| 最後巡檢時間 | 巡檢執行的時間 | 每次巡檢更新 |
| 頁面最後更新 | 含 commit hash | 有 commit push 才更新 |

**注意**：這兩個是不同的欄位，不可混淆。

## 未來改善

已規劃 Supabase 整合方案（見 `plan/aipmtw-dashboard-enhancement.md`），將所有硬編碼資料遷移至資料庫，前端動態讀取，消除每次巡檢改寫 HTML 的需要。

## 管理員注意事項

- 手動修改 index.html 中的資料物件後，下次巡檢會自動覆蓋
- 若需永久性修改，應修改巡檢邏輯而非直接改資料
- commit hash 遞迴更新是已知問題，Supabase 方案將根本解決
