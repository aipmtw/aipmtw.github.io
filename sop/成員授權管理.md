# SOP：成員授權管理

## 目的

管理 aipmtw 組織下各 repo 的 Collaborator 權限，決定誰可以透過 GitHub Issues 提交需求。

## 授權架構

```
GitHub 用戶提交 Issue / 功能申請
        ↓
巡檢系統檢查授權
        ↓
  ┌─────┼─────────────┐
  ↓     ↓             ↓
Owner  Collaborator   非授權用戶
  ↓     ↓             ↓
直接處理 直接處理      留言提示需 /approve
```

### 授權優先順序

1. **Repo Owner**：repo 擁有者（如 `aipmtw`），不會出現在 collaborator API 回傳中，需透過 `repos/{owner}/{repo}` 的 `owner.login` 欄位判斷
2. **Collaborator**：透過 `repos/{owner}/{repo}/collaborators/{username}` API 檢查（回傳 204 表示是 collaborator）
3. **非授權用戶**：以上皆非，需管理員 `/approve`

> **注意**：GitHub 的 collaborator API 不包含 repo owner，這是 GitHub API 的設計行為。外包協作時，我們的 repo 會由 owner（aipmtw）管理，外部團隊成員需加為 Collaborator 才有權限。

## Collaborator 管理

### 新增 Collaborator

```bash
# 透過 GitHub API
gh api -X PUT repos/aipmtw/<repo>/collaborators/<username> -f permission=push

# 或在 GitHub 網頁：Settings → Collaborators → Add people
```

### 查看 Collaborator 清單

```bash
gh api repos/aipmtw/<repo>/collaborators --jq '.[].login'
```

### 移除 Collaborator

```bash
gh api -X DELETE repos/aipmtw/<repo>/collaborators/<username>
```

## 授權流程

### 自動授權（透過申請空間）

1. 用戶在首頁申請空間
2. 管理員 `/approve` 後，GitHub Actions 自動將申請人加為新 repo 的 Collaborator（push 權限）

### 手動授權

管理員直接執行 API 或在 GitHub 網頁操作。

### 非 Collaborator Issue 處理

1. 巡檢系統偵測到非 Collaborator 的 issue
2. 自動留言提示（僅一次）
3. 管理員選擇：
   - `/approve` → 本次處理
   - 加為 Collaborator → 永久授權
   - 不處理 → issue 保持 open

## 權限等級

| 權限 | 說明 | 使用場景 |
|------|------|---------|
| push | 可讀寫程式碼 | 成員個人空間 |
| admin | 完整管理權限 | 僅限管理員 |

## 管理員注意事項

- 每人最多 2 個空間（首頁申請表單會檢查）
- 巡檢系統每次都即時查詢 collaborator 清單，不使用快取
- 新加入的 Collaborator 會在下次巡檢時自動被識別
